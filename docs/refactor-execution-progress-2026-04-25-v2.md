# 重构执行进展报告（第二阶段，2026-04-25）

> 基于上一版整改目标，继续推进架构收敛。本阶段重点不是继续做零碎微调，而是把页面层、控制层、导航层、搜索层进一步分开，避免 `GraphPage` 和 `useGraph` 再次膨胀。

## 本阶段已经完成的架构收敛

### 1. 页面层从“大控制器”继续收缩为装配层

`src/components/GraphPage.tsx` 已经从原先承载大量业务协调逻辑的页面组件，逐步转向真正的页面装配层：

- 负责页面布局
- 负责子组件组合
- 负责把状态与事件下发给子组件
- 不再直接承担大部分页面副作用

同时，右侧面板也已经独立出 `src/components/GraphRightPanel.tsx`，避免 `GraphPage` 继续堆叠右侧详情 / 样式切换逻辑。

### 2. 页面级副作用已下沉到 `useGraphPageController`

新增的 `src/hooks/useGraphPageController.ts` 负责聚合页面级副作用，包括：

- 页面进入日志
- 房间加载触发
- 搜索高亮触发
- dirty 状态同步
- tab 保存注册

这一步的效果是：

- `GraphPage` 不再直接挂大量 effect
- 页面级控制逻辑有了明确收口点
- 后续如果要替换页面实现，迁移成本明显降低

### 3. 导航与快捷键状态入口更统一了

`src/hooks/useNavContext.ts` 已经从单纯的只读适配器，向“统一导航状态入口”推进：

- `searchQuery` 由导航上下文统一读写
- `selectedNodeId` 由导航上下文统一读写
- `clearSelectedNode` 也已统一暴露

`src/hooks/useKeyboard.ts` 也已经不再直接操作 `tabStore`，而是通过导航上下文清理选中状态。

这意味着：

- 选中态的写入路径更统一
- 快捷键层不再直接依赖底层 store 细节
- UI 层和导航层的边界更清晰

### 4. `useGraph` 的导航职责已经拆出

`src/hooks/useGraph/navigation.ts` 已经把导航行为独立出来，包含：

- 返回上一层
- 按历史索引跳转
- 回到根级

这让 `useGraph` 本体更接近“图谱核心逻辑 hook”，而不是“图谱 + 导航 + 持久化 + 状态协调”的超大控制器。

### 5. 搜索高亮已抽成纯函数

`src/hooks/useGraph/search.ts` 里提供了纯函数 `applySearchHighlight`。

这意味着搜索高亮已经从 `useGraph` 主体中剥离出来，变成：

- 更容易测试
- 更容易复用
- 更容易替换的纯逻辑模块

---

## 当前还剩下的架构问题

### 1. `useGraph` 仍然较厚，但边界已经更清楚

尽管导航已独立，`useGraph` 仍然保留了不少职责：

- 图数据加载
- React Flow 事件处理
- 节点 / 边 CRUD 协调
- layout 持久化
- dirty 状态管理
- 选中态同步

这不是坏事，但说明它仍然是图谱领域的核心协调器，而不是极薄的 hook。

### 2. 双 store / 多 store 关系仍在，需要继续收敛

当前导航上下文虽然更统一了，但系统底层依然存在：

- `appStore`
- `tabStore`
- `roomStore`

这意味着双写和兼容分流仍需要继续谨慎清理，尤其是 Tab 模式与全局模式的语义边界。

### 3. `GraphPage` 还剩少量页面事件编排

`GraphPage` 现在已经明显变薄，但还有少量页面编排逻辑，比如：

- 右侧面板控制
- 快捷键接线
- 交互组合

这些已经属于合理范围，不建议为了“继续拆”而过度优化。

---

## 这次重构的关键结论

本阶段不是在继续做细碎的修补，而是完成了一个更重要的结构变化：

- 页面层从厚重控制器，转向装配层
- 副作用从页面下沉到 controller
- 导航从 `useGraph` 中剥离到独立模块
- 搜索高亮从 `useGraph` 中剥离为纯函数
- 选中态和搜索态的写入入口，开始朝统一方向收敛

这已经足以证明：当前优化方向是正确的，而且没有陷入无意义的过度拆分。

---

## 后续建议

如果后面继续推进，建议只做两类事情：

1. **收口而不是再拆分**
   - 继续保持当前边界清晰
   - 不要再把 `GraphPage` 拆得更碎
   - 不要再把 `useGraph` 切成过多微型 hook

2. **统一状态语义**
   - 继续收敛 Tab / 全局模式的状态定义
   - 明确哪些字段是单一事实来源
   - 逐步减少 store 之间的桥接

---

## 总结

这次重构已经把项目带到了一个更合理的架构阶段：

- 页面更薄了
- 控制层更清楚了
- 导航逻辑独立了
- 搜索逻辑纯函数化了
- 选中态入口更统一了

接下来更适合做的是“收口、验证、稳态维护”，而不是继续追求更细颗粒度的拆分。
