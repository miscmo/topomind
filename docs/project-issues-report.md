# 项目问题分析报告

> 依据项目目录下的 rule 约束，以及当前代码实现状态，对项目中已经暴露出的结构性问题做一次阶段性梳理。

## 一、结论摘要

当前项目的主要问题不是单一 bug，而是**页面层、导航层、状态层之间的职责边界不清**，并且已经出现了明显的**Tab / 全局模式兼容复杂化**、**跨 store 同步困难**、**副作用分散**、**可测试性不足**等问题。

如果继续沿着当前实现方式演进，后续会越来越容易出现：

- 逻辑重复
- 状态不同步
- 导航行为难以维护
- 组件越来越臃肿
- 新功能接入成本越来越高

---

## 二、依据规则得到的核心问题

项目规则强调了几件事：

- React / TypeScript 代码应清晰、可读、可维护
- 尽量使用单一职责、函数式、模块化设计
- 不要过度复杂化
- 不要把展示逻辑、状态派生、业务动作混在一起
- 尽量保持现有代码风格，不做无关重构

结合当前代码，这些原则并没有被很好地贯彻到一些关键模块中。

---

## 三、当前项目的主要问题

### 1. `GraphPage` 职责过重

`src/components/GraphPage.tsx` 同时承担了很多任务：

- 页面布局渲染
- Tab / 全局模式切换
- 图谱上下文挂载
- 右侧面板宽度拖拽
- 监听页面进入日志
- 维护搜索同步
- 维护选中节点同步
- 维护 room 状态恢复
- 处理加载房间逻辑
- 处理快捷键和上下文菜单

这已经不是“页面入口组件”，而是一个**高度聚合的业务控制中心**。

问题后果：

- 任何小改动都可能牵连多个行为
- 很难判断某个副作用属于谁
- 代码阅读成本很高
- 测试时很难只覆盖某一层逻辑

---

### 2. Tab 模式与非 Tab 模式的分支散落太多

当前代码中，Tab 逻辑与全局逻辑是分散在多个文件里的，而不是通过统一的数据模型封装。

例如：

- `GraphPage` 里根据 `tabId` 手工选择 `tabStore` 或 `roomStore`
- `useGraph` 里再次根据 `tabId` 决定写哪个 store
- 面包屑又单独处理一套路径来源
- `DetailPanel`、搜索、选中状态也存在兼容桥接

这意味着项目已经进入一种“**双状态源并行维护**”的阶段。

问题后果：

- 任意一个状态字段都可能有两个来源
- 同步逻辑容易漏
- 新功能要同时支持两套路径
- 代码中到处都是“兼容层”而不是“统一层”

---

### 3. 导航逻辑与 UI 组件耦合

从设计意图看，面包屑应该只是展示导航路径，但当前实现中，UI 组件很可能直接负责：

- 推导显示内容
- 判断根级 / 历史级 / 当前项
- 触发跳转
- 写日志
- 直接调用图谱导航方法

这类耦合会让 UI 组件变成“半个控制器”。

问题后果：

- 组件不够纯粹
- 很难复用
- 很难对路径计算和点击行为分别测试
- 一旦导航规则变化，UI 和业务要一起改

---

### 4. `useGraph` 过于庞大，已经接近“全功能控制器”

`src/hooks/useGraph.ts` 中同时包含了：

- 房间加载
- 布局持久化
- 节点/边 CRUD
- React Flow 事件处理
- 搜索高亮
- 导航前保存
- 返回上一层
- 跳转历史节点
- 回到根级
- 选中态同步
- dirty 状态管理

这使得 `useGraph` 变成了一个“**超大 hook**”。

问题后果：

- 很难定位某个行为的边界
- 很难拆出单元测试
- 任何导航逻辑改动都容易影响存储和加载
- 容易产生闭包、时序和副作用方面的隐患

---

### 5. 状态同步依赖过多“兼容桥接”

目前存在明显的“桥接写法”，比如：

- Tab 状态写入后，再同步到 `roomStore`
- `GraphPage` 中为了兼容旧逻辑，搜索词同时写全局 store 和 tab store
- 选中节点状态在 app store 与 tab store 之间来回同步
- `GraphPage` 还需要在合适时机把 tab 状态恢复到 `roomStore`

这说明系统缺少一个真正的“单一事实来源”。

问题后果：

- 写入路径多，读路径也多
- 数据最终是否一致，取决于每个同步点是否正确执行
- 一旦时序发生变化，就可能出现 UI 显示错乱

---

### 6. 副作用管理分散且时序复杂

当前代码中大量使用：

- `useEffect`
- `queueMicrotask`
- ref 兜底防止闭包过期
- 多处手动监听和清理

这说明项目已经在用很多手段避免 React 更新时序问题。

这不是坏事，但也说明副作用已经复杂到需要“打补丁式控制”。

问题后果：

- 可读性下降
- 维护者很难判断为什么必须这样写
- 未来修 bug 时容易误删关键保护

---

### 7. 类型和接口还不够统一

项目虽然使用 TypeScript，但部分地方仍然存在：

- 以 store 原始字段直接驱动 UI
- 业务状态没有抽象成标准模型
- 一些地方依赖隐式约定，而不是明确接口
- 某些逻辑仍然通过临时对象直接传递

这会让 TypeScript 的收益打折扣。

问题后果：

- 类型只能保证“字段存在”，不能保证“语义正确”
- 新字段加入后，很多地方要手动同步
- 不同模块对同一个概念的理解可能不一致

---

### 8. 可测试性不足

当前实现中最难测试的部分主要是：

- 路径推导
- Tab / 非 Tab 导航分流
- 历史栈跳转
- root 返回
- 当前状态与展示状态的映射

原因是这些逻辑被混合在 UI、hook、store 和副作用里。

问题后果：

- 单测难写
- 回归风险高
- 改一个地方，容易引发隐性问题

---

## 四、从当前实现中看到的具体风险点

### 风险点 1：导航行为和状态更新可能存在竞态

在 `useGraph` 中，加载、保存、跳转都夹杂异步过程，且大量依赖当前 nav state。
如果未来再增加更多异步操作，容易出现：

- 路径切换后，旧请求覆盖新请求
- roomStore 和 tabStore 先后写入顺序不一致
- 面包屑与实际加载内容短暂不一致

### 风险点 2：页面层组件越来越难继续加功能

如果继续在 `GraphPage` 中堆逻辑，新需求例如：

- 面包屑折叠
- 路径菜单
- 快捷导航
- 多窗口同步
- 多标签批量操作

都会让页面组件进一步膨胀。

### 风险点 3：兼容层会变成长期技术债

当前不少代码是在“兼容迁移期”语义下写的，但如果没有明确的收敛计划，兼容层会永久存在，最后变成真正的主逻辑。

---

## 五、优先级建议

### 高优先级

1. **拆分 `GraphPage` 的职责**
2. **为面包屑建立统一模型，避免直接消费原始 store 字段**
3. **收敛 Tab / 全局模式的数据读取方式**
4. **把导航行为从 UI 层中抽离出来**

### 中优先级

5. **把 `useGraph` 中的导航、加载、持久化进一步拆层**
6. **减少 store 之间的双向同步桥接**
7. **明确哪些 state 是单一事实来源**

### 低优先级

8. **整理类型模型和公共领域对象**
9. **补齐关键路径的单元测试**
10. **优化副作用时序处理，减少临时补丁式写法**

---

## 六、建议的短期改进方向

如果只做最小必要改动，建议按下面顺序推进：

1. 先抽出面包屑的数据模型和计算层
2. 再收敛导航入口，减少 UI 直接调用业务方法
3. 然后清理 `GraphPage` 中与展示无关的逻辑
4. 最后逐步降低 `useGraph` 的复杂度

这样可以在不破坏现有功能的前提下，慢慢把结构拉回到可维护状态。

---

## 七、最终判断

项目当前最大的问题不是“代码不能跑”，而是**架构已经开始被临时兼容逻辑拖着走**。

如果不尽快收敛：

- 新功能会越来越慢
- 回归 bug 会越来越多
- 维护成本会持续上升

如果先处理职责边界和统一数据模型，后续很多问题都会自然变简单。
