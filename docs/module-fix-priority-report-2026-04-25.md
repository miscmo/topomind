# 模块修复优先级报告（2026-04-25）

> 本报告基于当前已经拆分出来的模块，进一步给出“功能是否完整、是否存在缺失/异常、建议优先修复什么”的结论。

---

## 总体结论

这次拆分后的模块大部分已经可以正常工作，整体结构比之前更清晰。但如果从“模块功能完整性”和“逻辑风险”角度看，当前还存在几个**优先级较高**的问题，主要集中在：

1. `GraphPage` 与 `GraphRightPanel` 的受控状态流还没有完全收口
2. `GraphPage.tsx` 仍然存在 3 个无法定位的 lint warning，属于未解释的异常信号
3. `useGraph` 仍然较厚，且底层 store 依赖没有真正消失
4. `useNavContext` / `useKeyboard` 虽然接口统一了，但双源状态结构仍然存在
5. `useGraph/navigation.ts` 的职责边界是对的，但模式分流依然明显，需要保持谨慎

从修复角度来说，当前不适合继续大拆分，而应该优先修复以下两类问题：

- **明确的异常信号**，比如 lint warning、接口不对齐、状态流不一致
- **逻辑语义残留**，比如双源状态、重复清理路径、厚 hook 中的 store 依赖

---

## 优先级总览

### P0 必修

- 修正 `GraphPage.tsx` 的未定位 warning
- 对齐 `GraphPage` 与 `GraphRightPanel` 的受控状态流
- 确认 `useGraph` 的导航 API / 返回值命名没有语义错位

### P1 尽快修

- 核查 `useKeyboard` 的 Escape 清理是否存在重复执行或语义重叠
- 核查 `useNavContext` 是否需要进一步区分“领域写入”和“UI 清理”
- 检查 `GraphCanvas` 中的全局副作用是否有必要继续保留

### P2 后续优化

- 进一步减轻 `useGraph` 对底层 store 的直接依赖
- 如果未来继续扩功能，再考虑将部分领域逻辑抽成 selector / 纯函数

---

## 一、模块逐项修复建议

### 1. `src/components/GraphPage.tsx`

#### 功能完整性

基本完整，页面已能完成：

- 页面布局
- 画布装配
- 右侧面板装配
- 快捷键和右键菜单连接

#### 主要问题

##### 问题 1：存在 3 个未定位 warning

这是当前最明确的异常点。虽然没有精确行号，但它说明 `GraphPage` 仍有某种静态检查层面的残留问题。

##### 问题 2：它仍然承担页面装配 + 少量控制职责

这本身不是 bug，但说明 `GraphPage` 还不是完全无状态纯装配组件。

##### 问题 3：右侧面板状态虽然已受控化，但页面层仍负责整套传参

这是正确方向，但目前还没完全收口。

#### 建议修复

- 优先定位那 3 个 warning 的来源
- 确认 `GraphRightPanel` 的受控 props 已全部由页面层统一提供
- 避免继续把新页面逻辑塞回 `GraphPage`

#### 结论

`GraphPage` 现在**功能可用，但仍是最高优先级的检查对象**。

---

### 2. `src/components/GraphCanvas.tsx`

#### 功能完整性

完整，画布渲染和交互链路都在。

#### 主要问题

##### 问题 1：边点击仍直接切换右侧面板为样式页

这属于页面控制语义，不算 bug，但意味着画布层还不够“纯”。

##### 问题 2：空白区点击 / 双击仍直接调用全局 `clearSelection()`

这和 `useNavContext` 的选中态清理形成了重复路径。功能上没坏，但语义不够统一。

##### 问题 3：`window.__reactFlow` 仍然是全局调试副作用

这不是功能问题，但如果要稳定收口，后续应考虑去掉或明确保留用途。

#### 建议修复

- 保留当前功能，但不要再增加新的页面级语义到画布里
- 如果后续要进一步统一清理路径，可以把选择清理集中到统一入口

#### 结论

`GraphCanvas` 功能完整，没有明显缺失，但存在少量页面控制语义残留。

### 3. `src/components/GraphRightPanel.tsx`

#### 功能完整性

完整，结构比之前更清晰。

#### 主要问题

##### 问题 1：它已变成受控组件，但页面层必须传入更多状态

这是拆分后的正常结果，不是 bug；但它意味着 `GraphPage` 还必须维护面板状态。

##### 问题 2：内容切换逻辑仍然依赖 `rightPanelTab`

这也是正常的，但要避免未来再把业务逻辑塞回来。

#### 建议修复

- 只需要确保它持续保持“纯展示 + 事件回调”即可

#### 结论

功能完整，是当前拆分里较健康的模块之一。

### 4. `src/hooks/useGraphPageController.ts`

#### 功能完整性

完整，承担页面级副作用是合理的。

#### 主要问题

##### 问题 1：依赖 `nav` 快照

`const nav = getNavState()` 的写法意味着它是一个 render 快照，不是持久订阅。

这不算 bug，但如果后续状态更新更频繁，容易出现：

- effect 依赖不直观
- 某些字段变化看起来“跟着变了”，但实际是下一次 render 才反映

##### 问题 2：职责已经接近合理上限

它目前承载的东西已经不少了，后续不建议继续加新副作用。

#### 建议修复

- 暂时不改结构
- 只要保证后续不要再往里塞新职责即可

#### 结论

功能完整，当前风险不在缺失功能，而在后续继续膨胀。

---

### 5. `src/hooks/useNavContext.ts`

#### 功能完整性

完整，已经能承担导航上下文入口的角色。

#### 主要问题

##### 问题 1：底层仍然是双源分流

`tabId` 存在时操作 `tabStore`，否则操作 `roomStore` + `appStore`。

这意味着：

- 使用方式统一了
- 但状态源没有统一

##### 问题 2：`clearSelectedNode` 与 `setSelectedNodeId(null)` 的语义较接近

功能没错，但未来维护者可能会不清楚：

- 一个是领域写入
- 一个是 UI 语义清理

#### 建议修复

- 不必急着改接口
- 但建议后续统一文档化语义，避免误用

#### 结论

功能完整，但状态统一仍停留在包装层，不是真正底层收敛。

### 6. `src/hooks/useKeyboard.ts`

#### 功能完整性

完整，快捷键功能可用。

#### 主要问题

##### 问题 1：Escape 仍然做了两次清理

当前同时调用：

- `getNavState().clearSelectedNode()`

这会让清理逻辑出现语义重叠。

##### 问题 2：获取选中项依赖 `getNavState()` 快照

当前可用，但不是最强的一致性写法。

#### 建议修复

- 如果后面要进一步统一选中态，可以把清理逻辑收敛成单一入口
- 暂时不要拆得更细

#### 结论

功能完整，没有明显缺失，但清理路径仍然重复。

### 7. `src/hooks/useGraph.ts`

#### 主要问题

`getActiveSelectedNodeId` / `setActiveSelectedNodeId` 仍是底层双源模式的体现。

#### 建议修复

- 暂时不要再继续切得更碎
- 如果要修复，只修明显的逻辑错位和 API 名称语义问题

#### 结论

功能完整，但仍是后续维护最需要盯住的核心 hook。

## 三、最终建议

从模块完整性和 bug 风险角度看，当前最该做的是：

- **优先处理明确异常信号**
- **不要继续盲拆模块**
- **先把状态语义和接口语义收口**

如果继续做架构切分，收益已经开始下降；但如果先修复上面的异常点，当前这套拆分会更接近稳定可用的状态。
