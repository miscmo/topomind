# 重构执行进展报告（2026-04-25）

> 依据当前代码问题整改任务清单，先执行 P0 级别的最小安全修正，并同步记录当前仍存在的问题。

## 已执行的第一步

### 1. 修正了 `useGraph` 中一个明显无效的分支

`src/hooks/useGraph.ts` 里的 `navigateToRoom` 之前存在一个可疑条件：

- 先取出当前路径 `dirPath`
- 再用 `savedDirPath && savedDirPath !== dirPath` 判断是否保存
- 但 `savedDirPath` 和 `dirPath` 是同一个值，因此该分支实际上不会成立

我已经把这段逻辑改成了：

- 只要当前路径存在，就先执行保存
- 保留后续的导航与加载流程不变

这一步的意义是：

- 去掉了误导性的“假保护”代码
- 让行为更直观
- 降低后续维护者误判的风险

## 目前仍然存在的核心问题

### 1. 单一事实来源还没有真正统一

虽然 `Breadcrumb` 已经拆出 model / actions，但当前状态仍然分散在：

- `appStore`
- `tabStore`
- `roomStore`

`useNavContext` 仍然在根据 `tabId` 做兼容分流。

这说明：

- 导航与选中状态还没有真正统一
- 只是把部分桥接逻辑从页面层移走了
- 但源头上的“双写”问题仍旧存在

### 2. `useGraph` 仍然过重

`useGraph` 仍然同时承担：

- 加载
- 持久化
- React Flow 事件处理
- 导航
- 选中态同步
- 搜索高亮
- dirty 状态

这仍然是一个偏大的控制器 hook。

### 3. `GraphPage` 仍然偏厚

`GraphPage` 目前还是页面编排 + 副作用 + 兼容同步的组合体。

它已经比旧版本好一些，但还没有达到“薄页面层”的目标。

## 下一步建议

如果继续执行，我建议按下面顺序推进：

1. 先统一导航/选中状态的写入源
2. 再拆 `useGraph` 中的导航和持久化边界
3. 然后继续薄化 `GraphPage`
4. 最后再补测试和收敛兼容层

## 小结

这次我已经开始执行 P0，并先修掉了一个明确的无效分支。

但从整体上看，项目的架构性问题仍然存在，只是比之前更清楚地暴露出“应该先统一状态来源，再继续拆控制器”的方向。
