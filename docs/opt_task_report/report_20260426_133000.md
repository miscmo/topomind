## 一、补充发现的问题清单

### 1. 功能问题

#### 1.1 `tabStore` 与 `roomStore` 仍存在双源状态语义
- 位置：`src/stores/tabStore.ts`、`src/stores/roomStore.ts`
- 现象：Tab 模式下房间导航状态保存在 `tabStore`，非 Tab 模式保存在 `roomStore`，页面层通过 `useNavContext` 做桥接。
- 风险：功能可用，但语义上属于双源结构，维护时容易出现“只改一边”的问题。
- 验证：切换 Tab、进入房间、返回、恢复状态，检查 Tab 与全局状态是否一致。

#### 1.2 `tabStore.removeTab()` 对关闭后的活跃 Tab 选择是就近回退
- 位置：`src/stores/tabStore.ts`
- 现象：关闭当前活跃 Tab 时，会选择前一个 Tab，否则回到 home。
- 风险：行为合理，但如果关闭的是数组首项，fallback 邻近逻辑需要持续验证。
- 验证：关闭不同位置的 Tab，检查 activeTabId 是否符合预期。

#### 1.3 `roomStore.goBack()` 在历史栈清空后会回到全局视图
- 位置：`src/stores/roomStore.ts`
- 现象：当历史栈最后一个房间返回后，currentRoomPath 变为 null，currentRoomName 变为“全局”。
- 风险：功能合理，但依赖调用方正确恢复 kbPath。
- 验证：连续进入多个房间后逐级返回，检查是否正确回到全局视图。

#### 1.4 `storage.saveGraphDebounced()` 的防抖保存依赖 buildMetaFn 闭包
- 位置：`src/core/storage.ts`
- 现象：保存时通过闭包生成 meta，再写入。
- 风险：若闭包引用的状态不是最新，可能保存旧数据；不过当前设计上配合 refs 使用基本可用。
- 验证：频繁编辑节点后观察保存结果是否与 UI 一致。

### 2. 逻辑 Bug

#### 2.1 `roomStore.goToRoom()` 的分支逻辑较复杂
- 位置：`src/stores/roomStore.ts`
- 现象：既支持 history 中跳转，也支持不在 history 时直接 enterRoom。
- 风险：虽然功能完整，但分支较多，后续若改历史机制容易引入回退错误。
- 验证：对历史项与非历史项分别调用 goToRoom，观察导航结果。

#### 2.2 `tabStore.goBackInTab()` 在新 history 为空时回到 KB 全局视图
- 位置：`src/stores/tabStore.ts`
- 现象：与 `roomStore.goBack()` 的语义接近，但是在 Tab 内部实现。
- 风险：逻辑正确，但要注意 currentRoomPath 与 kbPath 的一致性。
- 验证：在 tab 模式反复进入/返回，检查面包屑与页面状态。

#### 2.3 `storage.readLayout()` / `saveLayout()` 都依赖 `normalizeMeta()`
- 位置：`src/core/storage.ts`
- 现象：元数据在读写时都被标准化。
- 风险：如果 normalize 规则与前端期待不一致，可能出现边属性丢失或默认值偏移。
- 验证：保存后重新读取，检查 edges / zoom / pan / canvasBounds 是否保留。

### 3. 安全风险

#### 3.1 `storage.writeConfig()` 会把合并后的配置整体写回
- 位置：`src/core/storage.ts`
- 现象：使用 cachedConfig 与传入 config 合并后写回。
- 风险：如果上层把敏感字段也塞入配置对象，会被整体持久化。
- 验证：检查工作目录配置内容是否只包含预期字段。

#### 3.2 `storage.loadImage()` 会将读取到的二进制转成 Blob URL
- 位置：`src/core/storage.ts`
- 现象：图片加载时生成 `blob:` URL 并注册到回收表。
- 风险：这是正常行为，但如未正确回收可能造成泄漏。
- 验证：反复打开/关闭含图片卡片，检查回收是否生效。

#### 3.3 存储层直接委托 `FSB`，权限边界依赖底层实现
- 位置：`src/core/storage.ts`
- 现象：所有文件系统操作由 FSB 承担。
- 风险：如果 FSB 本身的路径校验不足，前端会被间接放大。
- 验证：路径越界、非法名称、重复文件操作是否被拒绝。

### 4. 内存泄漏

#### 4.1 `_saveManager` 防抖定时器需要确保在清除/退出时被回收
- 位置：`src/core/storage.ts`
- 现象：`saveGraphDebounced()` 会持有定时器。
- 风险：当前有 `clearTimer/clearAll`，但如果调用链遗漏，仍可能挂住。
- 验证：反复快速保存、切换页面、退出应用后检查定时器是否释放。

#### 4.2 `_imageUrlRegistry` 的 revokeAll 依赖应用卸载调用
- 位置：`src/core/storage.ts`、`src/App.tsx`
- 现象：图片 URL 在应用卸载时统一回收。
- 风险：若未来引入局部销毁路径，可能只在整体卸载才回收。
- 验证：含图片文档反复切换，检查 URL 数量是否持续增长。

#### 4.3 `tabStore` / `roomStore` 均为全局单例，历史状态不自动过期
- 位置：`src/stores/tabStore.ts`、`src/stores/roomStore.ts`
- 现象：这是设计选择，但意味着状态长期驻留内存。
- 风险：在大量 Tab / 长会话场景下，可能积累状态。
- 验证：长时间运行和频繁打开关闭 Tab 后检查内存变化。

### 5. 体验问题

#### 5.1 `tabStore` 与 `roomStore` 双源结构对维护者认知成本高
- 位置：`src/stores/tabStore.ts`、`src/stores/roomStore.ts`
- 现象：状态看似统一，但实际上分散在两个 store。
- 风险：后续修复或扩展时容易误改。
- 验证：阅读切换/返回/保存相关代码。

#### 5.2 `storage.createCard()` 遇到重复同名时直接抛错
- 位置：`src/core/storage.ts`
- 现象：已有同级同名卡片时会 throw。
- 风险：功能上正确，但上层要保证能把错误转成友好提示。
- 验证：尝试创建同名卡片，检查 UI 提示是否清晰。

#### 5.3 `storage.readMarkdown()` / `writeMarkdown()` 依赖固定 README.md 命名
- 位置：`src/core/storage.ts`
- 现象：文档文件固定为 README.md。
- 风险：功能稳定，但未来若改命名规则需同步所有调用点。
- 验证：卡片文档读写流程正常。

### 6. 工程化问题

#### 6.1 `storage.ts` 文件较长，职责集中度仍然偏高
- 位置：`src/core/storage.ts`
- 现象：配置、目录、卡片、布局、图片、工具都在一个文件里。
- 风险：后续继续扩展会增大维护成本。
- 验证：阅读文件结构即可感知。

#### 6.2 配置读写没有更细粒度 schema 校验
- 位置：`src/core/storage.ts`
- 现象：通过 normalize 做基本规范化，但不是强 schema。
- 风险：配置字段扩展时容易出现隐式兼容。
- 验证：给配置文件注入异常字段，观察是否被正确忽略。

### 7. TS 类型问题

#### 7.1 `storage.ts` 仍存在较多 `unknown` / 强制断言
- 位置：`src/core/storage.ts`
- 现象：normalize 阶段大量使用 `unknown` 和断言。
- 风险：类型保护不足。
- 验证：查看编译和边界数据处理结果。

#### 7.2 `tabStore` 与 `roomStore` 对外暴露的返回类型是高度自定义结构
- 位置：`src/stores/tabStore.ts`、`src/stores/roomStore.ts`
- 现象：API 一致，但类型层面较复杂。
- 风险：后续调用时容易出现误用。
- 验证：检查各调用点是否都通过正确类型访问。
