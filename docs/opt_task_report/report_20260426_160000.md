## 一、发现的问题清单

### 1. 功能问题
1. `log-backend` 功能完整，支持日志写入、查询、订阅、取消订阅、级别设置等接口。
2. `logger` 功能完整，提供 console + 文件双写日志机制。
3. `close-guard` 功能完整，能注册 Tab 保存器并在退出前 flush 脏 Tab。

### 2. 逻辑 Bug
1. `logSubscribe` / `logUnsubscribe` 依赖 `_listeners` 和 `_ipcRegistered` 两个状态，逻辑正确，但若调用顺序异常，可能出现“已取消监听但 IPC handler 仍在”的边界疑问。
2. `close-guard.flushTabs()` 在 saver 失败时立即返回，行为合理，但会中断后续 Tab 的保存，属于预期但需明确的失败策略。
3. `logger.catch()` 会记录 stack 到文件日志，功能正确，但若错误中包含敏感路径或数据，需注意日志体积与敏感信息。
4. `close-guard` 通过 `window.__topomindCloseGuard` 暴露守卫对象，能工作，但属于显式全局桥接点。

### 3. 安全风险
1. `log-backend` 和 `logger` 会写入文件日志，若 params 包含敏感信息，日志系统可能放大泄露面。
2. `close-guard` 暴露到 `window` 的守卫对象理论上可被页面脚本访问，虽然作用面有限，但仍属于全局暴露。
3. `logSubscribe` 的 IPC 订阅依赖主进程只发可信日志事件，若事件来源不受控，监控页会显示异常数据。
4. `logger.catch()` 和 `logWrite` 的组合需要继续避免将用户私密内容直接写入日志。

### 4. 内存泄漏
1. `logSubscribe` 维护的 `_listeners` 集合若调用方不 `logUnsubscribe`，会持续增长。
2. `close-guard` 依赖 Tab saver 注册/注销的对称性，若组件卸载时未注销保存器，Map 可能残留。
3. `logger` 在每次记录时都会尝试写文件日志，若失败并重试逻辑扩展不当，需持续注意内存与队列堆积。

### 5. 体验问题
1. `log-backend` 功能完整，但日志写入与订阅在调用方看来较底层，错误处理体验主要取决于上层。
2. `close-guard` 在退出前 flush 全部脏 Tab，体验合理，但若某个 Tab 保存失败，会直接中断整批操作，用户需得到清晰反馈。
3. `logger` 记录的信息较完整，适合排障，但如果日志量大，监控页可读性要持续关注。

### 6. 工程化问题
1. `log-backend`、`logger`、`close-guard` 都属于全局基础设施，应尽量保持接口稳定。
2. `close-guard` 目前通过全局变量暴露给渲染层，工程上可用，但后续需要持续审视是否保留这种桥接方式。
3. `logger` 的 console/file 双写模式较实用，但所有日志都汇入文件，后续需关注日志量控制。

### 7. TS 类型问题
1. `log-backend` 里 `LogEntry`、`LogQueryOptions` 类型是内部定义，和监控页类型存在结构联动风险。
2. `close-guard` 暴露给 window 的对象通过扩展类型写入，类型可用但偏手工。
3. `logger.catch()` 和 `logWrite()` 调用链中仍有不少 `unknown` / `any` 边界。

## 四、功能优化详情

### 问题 1：logSubscribe / logUnsubscribe 依赖双状态协同
- 问题描述：监听器集合与 IPC 注册状态需保持一致。
- 原因分析：这是订阅式设计的典型实现。
- 解决方案：保持现状，但持续验证订阅/退订是否对称。
- 验证方式：反复打开/关闭监控页，检查日志是否重复推送。

### 问题 2：close-guard 的失败即中断策略
- 问题描述：任一 saver 失败会停止后续 flush。
- 原因分析：为了保证失败可见，采取了短路策略。
- 解决方案：当前合理；若未来需要更强容错，可改为收集失败列表。
- 验证方式：人为制造一个 saver 失败，看退出提示是否准确。

### 问题 3：window.__topomindCloseGuard 全局暴露
- 问题描述：关闭守卫对象挂在 window 上。
- 原因分析：用于主进程在退出前直接调用渲染层守卫。
- 解决方案：保持当前实现，但需持续关注安全边界。
- 验证方式：主进程退出前 flush 能正常触发。
