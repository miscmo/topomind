## 一、发现的问题清单

### 1. 功能问题
1. `usePageLogging()` 只在 `view === 'graph'` 时记录进入图谱页，符合当前需求，但如果未来增加更多视图状态，日志粒度需要重新评估。
2. `useRoomLoader()` 依赖 `queueMicrotask` 延后加载，当前用于规避竞态，但其行为对后续维护者不够直观。
3. `useTabDirtySync()` 仅在 Tab 模式下启用，符合设计，但其脏状态写回是全局 store 调用，容易与其它脏标记入口重叠。
4. `useGraph/navigation.ts` 的导航逻辑完整，但仍依赖底层 store 分流，属于兼容式实现。

### 2. 逻辑 Bug
1. `useRoomLoader()` 的 `queueMicrotask` 与 `isCreatingRef` 联动依赖时序，若后续有新的异步加载入口，可能出现“先被跳过、后又重复加载”的边界情况。
2. `useTabDirtySync()` 使用 `setTabDirtyRef.current!(tabId, isModified)` 的非空断言，当前可用，但如果 effect 生命周期被改写，容易出现空引用风险。
3. `useGraph/navigation.ts` 中 `navigateToRoom()` 在全局模式下先调用 `navigateToHistoryIndex(index)`，再读取 `getActiveNavState()` 加载，逻辑正确但耦合较深。

### 3. 安全风险
1. `usePageLogging()` 记录的日志参数包含当前路径与 tabId，属于业务日志信息；目前看不涉及敏感凭据，但日志结构应持续审视。
2. `useRoomLoader()` 与 `useTabDirtySync()` 均会触发全局状态写入，后续若增加外部输入，需确保不会引入状态污染。

### 4. 内存泄漏
1. `useRoomLoader()` 使用 `loadRoomRef` 和 `queueMicrotask`，当前未见明显泄漏，但 microtask 若连续堆积，仍需验证卸载时不会遗留无效任务。
2. `useTabDirtySync()` 订阅 `onDirtyChange` 并依赖返回的 cleanup，当前形式正确，但在多次切换 Tab 时仍需确认订阅被正确销毁。

### 5. 体验问题
1. `usePageLogging()` 的日志粒度较粗，仅记录“进入图谱页”，后续如果需要排查特定视图问题，可能不够细。
2. `useRoomLoader()` 的加载触发日志较多，若页面切换频繁，会产生日志噪音。
3. `useTabDirtySync()` 的脏状态同步对用户是透明的，但若脏标识变化与保存状态不同步，体验会受影响。

### 6. 工程化问题
1. `useRoomLoader.ts` 目前同时承担“触发房间加载”和“规避竞态”的责任，文件职责虽小但逻辑密度较高。
2. `useTabDirtySync.ts` 中使用 `useRef` 包装 setter 的写法在当前可用，但读者理解成本高。
3. `usePageLogging.ts`、`useRoomLoader.ts`、`useTabDirtySync.ts` 依赖全局 store/日志模块，后续扩展时需保持职责边界清晰。

### 7. TS 类型问题
1. `useRoomLoader()`、`useTabDirtySync()` 中仍有较多对函数签名的显式约定，类型上虽然成立，但对后续重构敏感。
2. `useTabDirtySync()` 的 `setTabDirtyRef` 采用 `useRef<(tabId: string, isDirty: boolean) => void>()` 的形式，类型可用但不够简洁。
3. `usePageLogging()` 的 `view` 类型是 `string`，如果后续视图枚举增加，建议改成更严格的联合类型。

## 四、功能优化详情

### 问题 1：房间加载使用 queueMicrotask 规避竞态
- 问题描述：`useRoomLoader()` 通过 microtask 延迟执行加载。
- 原因分析：这是为避免路径变化与创建流程竞态而引入。
- 解决方案：保持当前逻辑，但后续若新增加载入口要统一复用。
- 验证方式：快速切换房间、创建节点后观察是否重复加载或漏加载。

### 问题 2：脏状态同步依赖非空断言
- 问题描述：`useTabDirtySync()` 的 setter ref 使用非空断言调用。
- 原因分析：当前生命周期下通常成立，但结构上不够稳妥。
- 解决方案：后续可去掉 ref 包装或改成更直接的闭包写法。
- 验证方式：多次切换 Tab、关闭 Tab，确认脏标记仍同步正确。

### 问题 3：页面日志粒度偏粗
- 问题描述：`usePageLogging()` 只记录进入图谱页。
- 原因分析：当前够用，但诊断能力有限。
- 解决方案：如果后续需要更细排查，可增加更多事件标签。
- 验证方式：观察日志系统是否足以支持问题定位。
