## 【四、功能优化详情】

### 4.1 Bug Fix: fs-backend.ts 批量 TS2322 类型错误

**问题描述**：`_call` 函数返回 `Promise<unknown>`，16 个方法直接赋值给具体返回类型，导致 TS2322 "Type 'unknown' is not assignable to type 'X'" 批量错误。

**根因分析**：`window.electronAPI.invoke` 返回 `Promise<unknown>`，每个方法包装层都需要显式类型断言告诉 TypeScript 实际类型。

**解决方案**：
```typescript
// 修复前
listChildren: (parentPath) => _call('fs:listChildren', parentPath),
// 修复后
listChildren: (parentPath) => _call('fs:listChildren', parentPath) as Promise<FSBChildInfo[]>,
```

**验证方式**：`tsc --noEmit` → 0 errors ✅

---

### 4.2 Bug Fix: log-backend.ts Promise<unknown> 和 const 重赋值

**问题描述**：所有 7 个导出函数中 `_call(...)` 返回 `Promise<unknown>`，无法赋值给具体返回类型；同时 `_ipcHandler` 用 `const` 声明但后续被重新赋值导致 TS2588。

**根因分析**：
1. `_call` 返回类型为 `Promise<unknown>`，需要类型断言
2. `_ipcHandler` 先声明为 `const`，后续在 `logSubscribe`/`logUnsubscribe` 中被重新赋值

**解决方案**：
```typescript
// 类型断言
export async function logWrite(entry: Partial<LogEntry>): Promise<boolean> {
  return await (_call('log:write', entry) as Promise<boolean>)
}

// const → let
let _ipcHandler: ((entry: unknown) => void) | null = null
```

**验证方式**：`tsc --noEmit` → 0 errors ✅

---

### 4.3 Bug Fix: useNodeActions.ts TS18048 "possibly undefined"

**问题描述**：`const current = edge.data ?? ({} as KnowledgeEdge['data'])` 中 `{} as X` 的类型推导导致 `current` 在访问属性时仍被认为可能 `undefined`。

**根因分析**：`as` 类型断言在 TypeScript 严格模式下，左侧表达式可能被认为仍然是原类型（`EdgeData | {}`），需额外加非空断言。

**解决方案**：
```typescript
// 修复前
const current = edge.data ?? ({} as KnowledgeEdge['data'])
lineMode: current.lineMode ?? 'smoothstep'  // TS18048: current possibly undefined

// 修复后
const current = (edge.data ?? {}) as KnowledgeEdge['data']
lineMode: current?.lineMode ?? 'smoothstep'  // 可选链
```

**验证方式**：`tsc --noEmit` → 0 errors ✅

---

### 4.4 Bug Fix: graphOperations.ts 边类型展开不兼容

**问题描述**：
1. `...e.data` 展开后 `weight` 变成 `EdgeWeight | undefined`，与接口要求的 `EdgeWeight` 不兼容
2. 复杂 return 对象通过多个 `...` 展开后 TypeScript 无法推导最终类型

**根因分析**：TypeScript 结构类型展开时，所有可选字段保持可选状态，无法自动合并为必需字段。

**解决方案**：
```typescript
// 显式重新赋值必需字段
data: {
  ...e.data,
  relation: e.data?.relation ?? '相关',   // 确保非 undefined
  weight: e.data?.weight ?? 'minor',       // 确保非 undefined
} as KnowledgeEdge                          // 强制类型
```

**验证方式**：`tsc --noEmit` → 0 errors ✅

---

### 4.5 Bug Fix: graphBuilder.ts 类型注解缺少可选字段

**问题描述**：`childInfo` 参数注解为 `{ name: string }`，但实际对象含 `x?`/`y?` 可选字段，导致 4 处 TS2741 "Property 'x' is optional in type..." 错误。

**根因分析**：`GraphMeta.children` 中每个 child 的类型定义包含 `{ name: string; x?: number; y?: number }`，但调用点类型注解不完整。

**解决方案**：
```typescript
return normalizedChildren.map(([childName, childInfo]: [
  string,
  { name: string; x?: number; y?: number },  // 补全可选字段
], i) => { ... })
```

**验证方式**：`tsc --noEmit` → 0 errors ✅

### 4.6 Bug Fix: useStorage.ts 重复属性定义 TS1117

**问题描述**：`readConfig` 和 `writeConfig` 在 Store 对象字面量中被定义了两次（第一次在行 50-51，第二次在行 70-71），导致 TS1117 "Duplicate identifier 'readConfig'"。

**根因分析**：提取方法时保留了重复定义，未删除旧的占位符条目。

**解决方案**：删除第一次的错误定义，仅保留末尾正确的完整定义。

**验证方式**：`tsc --noEmit` → 0 errors ✅

### 4.7 Bug Fix: GraphCanvas.tsx Node 泛型不匹配

**问题描述**：`node as Node` 的 `Node` 类型未指定泛型参数，与 `KnowledgeNode`（实为 `Node<KnowledgeNodeData>`）不匹配。

**根因分析**：`@xyflow/react` 的 `Node` 是泛型类型，需要显式指定数据参数。

**解决方案**：
```typescript
import type { KnowledgeNode } from '../types'
// ...
node as KnowledgeNode  // 匹配 Node<KnowledgeNodeData>
```

**验证方式**：`tsc --noEmit` → 0 errors ✅

### 4.8 Bug Fix: ContextMenu.tsx 8 处空值传递 TS2322

**问题描述**：`targetId: string | null` 类型的变量传递给期望 `string` 的函数参数，导致 8 处 TS2322。

**根因分析**：右键菜单中部分边缘操作的目标 ID 可能为空，但 IPC 调用需要非空路径。

**解决方案**：
```typescript
// 添加空值守卫
action: async () => {
  if (targetId) await onEdgeStyle(targetId)
  onClose()
},

// 兜底空字符串
nodeId: paneTargetId ?? ''
```

**验证方式**：`tsc --noEmit` → 0 errors ✅

### 4.9 Bug Fix: graph.ts KnowledgeEdge 缺少 selected 字段

**问题描述**：`updateEdgeStyle` 中 `style.selected` 被赋值，但 `KnowledgeEdge` 数据类型未定义 `selected` 字段。

**根因分析**：`selected` 是运行时 UI 状态（选中高亮），非持久化字段，类型定义遗漏。

**解决方案**：
```typescript
export type KnowledgeEdge = Edge<{
  // ...existing fields...
  /** 选中状态（运行时状态，不持久化） */
  selected?: boolean
}>
```

**验证方式**：`tsc --noEmit` → 0 errors ✅
