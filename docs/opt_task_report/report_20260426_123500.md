# 项目优化补充报告（2026-04-26 12:35）

> 本补充报告基于 `docs/opt_task.md` 的要求，对 Electron 进程边界、安全、工程化与类型化问题做进一步检查。

## 一、补充发现的问题清单

### 1. 功能问题

#### 1.1 `send` 与 `sendSync` 的白名单策略不对称
- 位置：`electron/preload.js`
- 现象：`send` 现在使用的是 `ALLOWED_SEND_SYNC_CHANNELS` 白名单，而不是独立的 `ALLOWED_SEND_CHANNELS`。
- 风险：当前只有 `save:layout` 被允许同步与异步发送，若未来新增仅允许异步发送的通道，会因共用白名单产生误判。
- 验证：查看渲染进程是否需要调用除 `save:layout` 外的 `send` 通道。

#### 1.2 `GraphPage` 右键边的联动链路依赖页面层状态同步
- 位置：`src/components/GraphPage.tsx`、`src/components/GraphCanvas.tsx`
- 现象：画布只负责抛事件，页面层负责切换样式页并写入 `selectedEdgeId`。
- 风险：功能目前可用，但对页面状态的依赖仍然集中。
- 验证：右键边，确认样式页、选中边、菜单状态同时正确。

---

### 2. 逻辑 Bug

#### 2.1 `send` 白名单引用变量命名与实际用途不一致
- 位置：`electron/preload.js`
- 现象：`send` 函数检查 `ALLOWED_SEND_SYNC_CHANNELS`。
- 风险：这不是立即运行错误，但命名不清晰，后续维护容易误改。
- 验证：审查 `send` 实际可调用的通道是否符合设计。

#### 2.2 监听通道与发送通道职责分离不够明确
- 位置：`electron/preload.js`
- 现象：同一份白名单逻辑同时服务 invoke/send/sendSync/on/off。
- 风险：如果后续扩展更多 IPC 调用方式，维护复杂度会增大。
- 验证：检查新增 IPC 时是否能准确落到正确白名单。

---

### 3. 安全风险

#### 3.1 `preload.js` 仍是纯 JS，缺少编译期约束
- 位置：`electron/preload.js`
- 现象：IPC 白名单全靠手工维护，且是 JS 文件。
- 风险：新增通道时容易遗漏白名单或误放行。
- 验证：新增一个未授权通道，必须被拒绝。

#### 3.2 IPC 白名单集中度高，错误修改会影响整层通信
- 位置：`electron/preload.js`
- 现象：所有对外能力都由一个文件控制。
- 风险：该文件是 Electron 安全边界关键点，需要持续审计。
- 验证：逐条检查允许通道是否合理。

#### 3.3 render 进程通过 `window.electronAPI` 直接暴露能力
- 位置：`electron/preload.js`
- 现象：这是设计上允许的，但属于核心安全面。
- 风险：如果渲染层出现 XSS，攻击面集中在 `electronAPI`。
- 验证：确认所有高风险操作通道都在白名单内。

---

### 4. 内存泄漏

#### 4.1 `listenerMap` 采用 `WeakMap` 管理回调映射，整体方向正确，但需要持续确认解绑
- 位置：`electron/preload.js`
- 现象：`on/off` 通过 `WeakMap` 去重并解绑。
- 风险：如果渲染层忘记调用 `off`，仍可能积累监听器。
- 验证：切换页面 / 反复挂载卸载后，监听数量不应持续增长。

#### 4.2 `App.tsx` 持有多个全局订阅与清理入口
- 位置：`src/App.tsx`
- 现象：窗口事件、退出前保存、Blob URL 回收等集中在根组件。
- 风险：后续再加监听时，若遗漏清理容易泄漏。
- 验证：反复进入/退出页面并检查内存与事件监听。

---

### 5. 体验问题

#### 5.1 `App.tsx` 的启动状态切换依赖窗口 hash
- 位置：`src/App.tsx`
- 现象：`#/monitor` 用 hash 判断监控页。
- 风险：虽可用，但路由语义隐式，后续扩展不直观。
- 验证：修改 hash 后页面切换是否符合预期。

#### 5.2 `GraphCanvas` 的空白点击清理仍直接依赖全局 store
- 位置：`src/components/GraphCanvas.tsx`
- 现象：空白点击/双击直接执行全局清理。
- 风险：一致性可接受，但语义不够统一。
- 验证：对比 Escape、空白点击、切 Tab 的清理结果是否一致。

---

### 6. 工程化问题

#### 6.1 报告目录名称在文档中存在拼写不一致
- 位置：`docs/opt_task.md`
- 现象：文档中写的是 `opt_task_repot`，实际目录为 `opt_task_report`。
- 风险：容易误导后续执行路径。
- 验证：确认所有报告都统一输出到 `docs/opt_task_report`。

#### 6.2 Electron 相关 JS 文件缺少统一类型检查
- 位置：`electron/main.js`、`electron/preload.js`、`electron/*`
- 现象：主进程/预加载为 JS。
- 风险：工程一致性弱于 TS 层。
- 验证：查看构建与运行是否正常。

---

### 7. TS 类型问题

#### 7.1 `electron/preload.js` 无类型约束
- 位置：`electron/preload.js`
- 现象：通道字符串、回调签名、参数列表全为运行时校验。
- 风险：可维护性低。
- 验证：检查新增 IPC 通道是否严格通过白名单。

#### 7.2 `GraphCanvas` 仍然存在局部 `as any` 强转
- 位置：`src/components/GraphCanvas.tsx`
- 现象：React Flow 节点上下文菜单处仍有 `as any`。
- 风险：类型保护不完整。
- 验证：查看该处类型能否进一步收敛。

## 二、补充的架构优化目标

- 继续保持页面层、控制层、画布层分离
- 强化 Electron IPC 白名单的可审查性
- 避免 `send` / `sendSync` / `invoke` 的通道策略混淆
- 减少根组件中隐式全局订阅负担
