## 一、发现的问题清单

### 1. 功能问题
1. `session-reset` 能清理监控页、房间、Tab、应用状态并回收图片 URL，功能完整。
2. `HomePage` 功能完整，支持知识库列表、打开、删除、重命名、新建、导入、切换工作目录等主入口操作。
3. `HomePage` 内部同时承担列表加载、弹窗状态、右键菜单、工作目录、hover 日志、创建/导入流程等功能，复杂度偏高但当前可用。

### 2. 逻辑 Bug
1. `session-reset` 会先回收图片 URL，再重置多个 store，逻辑合理；但如果未来新增 store，必须记得同步加入重置链路。
2. `HomePage` 打开已有知识库时，会同时更新 `tabStore` 与 `roomStore`，这与多 Tab 架构一致，但状态桥接较复杂，后续要避免只更新其中一边。
3. `HomePage` 的右键菜单关闭逻辑依赖外部点击判断，当前可用，但如果 DOM 结构变化，菜单关闭边界需要重新验证。
4. `HomePage` 的导入流程依赖 `selectWorkDirCandidate` 返回结构与后端一致，若返回结构变更，导入路径填充可能失效。

### 3. 安全风险
1. `HomePage` 允许删除/重命名/导入知识库，属于高风险操作，必须继续依赖底层路径校验与确认弹窗。
2. `session-reset` 直接清空全局状态，若在不适当时机触发，会导致当前 UI 状态丢失。
3. `HomePage` 中的用户输入（知识库名）需要持续确保不会被直接用于不安全路径拼接。
4. `HomePage` hover 日志、创建日志、导入日志会进入全局日志系统，需避免写入敏感路径或个人信息。

### 4. 内存泄漏
1. `HomePage` 使用 `enterLogTimers` 管理 hover 防抖定时器，并在卸载时清理，当前设计正确。
2. `HomePage` 的外部点击监听在菜单可见时启用，并在清理函数中注销，当前没有明显泄漏。
3. `session-reset` 不持有长生命周期订阅，内存风险较低，但重置链路应继续保持幂等。

### 5. 体验问题
1. `HomePage` 的交互很多，作为首页功能完整，但新用户可能需要较多步骤理解新建/导入/打开/右键菜单。
2. `HomePage` 中“新建知识库”“导入知识库”两个入口都可用，建议后续继续保持文案与行为一致。
3. `session-reset` 属于后台行为，用户感知应该尽量平滑，避免中途状态丢失。

### 6. 工程化问题
1. `HomePage` 文件较长，功能密度很高，属于后续持续关注的工程化对象。
2. `session-reset` 的重置链路应保持与 store 增加同步更新。
3. `HomePage` 与多个 store / hook / storage 的联动较多，后续改动需格外谨慎。

### 7. TS 类型问题
1. `HomePage` 中存在少量显式断言与事件类型依赖，但当前可用。
2. `session-reset` 很简单，类型问题较少。
3. `HomePage` 的 KBItem/右键菜单类型为局部定义，功能上可用，但后续若共享范围扩大，可考虑提取到公共类型。

## 四、功能优化详情

### 问题 1：HomePage 功能密度高
- 问题描述：列表、弹窗、菜单、导入、切换工作目录都集中在一个页面。
- 原因分析：首页是产品主入口，天然复杂。
- 解决方案：当前可用；若后续继续扩展，建议按职责拆分。
- 验证方式：打开、删除、重命名、新建、导入流程都应正常。

### 问题 2：session-reset 需要同步新增 store
- 问题描述：新增全局状态后需记得纳入重置链路。
- 原因分析：重置逻辑集中在一个函数里。
- 解决方案：后续新增 store 时同步更新该函数。
- 验证方式：执行 reset 后状态必须全部回到初始值。

### 问题 3：HomePage 的导入/创建流程依赖底层校验
- 问题描述：上层输入必须最终由底层保证合法。
- 原因分析：路径与文件操作属于高风险域。
- 解决方案：保持现有确认和错误处理，并继续依赖底层校验。
- 验证方式：非法名称/路径输入应被拒绝并给出错误提示。
