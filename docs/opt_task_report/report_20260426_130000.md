## 一、补充发现的问题清单

### 1. 功能问题

#### 1.1 监控窗口使用 hash 路由分流，但渲染层仍需手工识别
- 位置：`src/App.tsx`、`electron/main.js`
- 现象：`#/monitor` 由根组件判断并渲染监控页，主进程拼接 hash 打开监控窗口。
- 风险：逻辑能用，但路由语义隐式，后续如果 hash 扩展容易冲突。
- 验证：打开监控窗口、主窗口，确认 hash 切换不互相影响。

#### 1.2 `before-quit` 的异步保存流程需要严格确认没有卡死窗口
- 位置：`electron/main.js`
- 现象：退出前先触发渲染层 flush，再决定是否真正退出。
- 风险：一旦渲染层 flush 卡住，主进程退出流程可能延迟。
- 验证：模拟 dirty 状态退出，确认不会出现无限等待。

### 2. 逻辑 Bug

#### 2.1 `before-quit` 依赖 `_isQuittingAfterFlush` 状态避免重复退出
- 位置：`electron/main.js`
- 现象：`before-quit` 中会先 preventDefault，再 flush，再调用 `app.quit()`；靠 `_isQuittingAfterFlush` 避免第二次拦截。
- 风险：当前逻辑可用，但如果未来退出路径再增加分支，状态标志容易被误用。
- 验证：多次触发退出、切换工作目录，确认不会重复弹框或重复 flush。

#### 2.2 `monitorWin` 关闭后手工置空，若窗口生命周期异常可能出现状态不同步
- 位置：`electron/main.js`
- 现象：`closed` 回调和 `toggleMonitorWindow()` 中都可能清空引用。
- 风险：正常可用，但需要注意避免重复 destroy / null 造成状态判断歧义。
- 验证：连续打开/关闭监控窗口，观察日志与引用状态。

### 3. 安全风险

#### 3.1 主进程内注册的 IPC 通道数量较多，安全边界集中
- 位置：`electron/main.js`、`electron/preload.js`
- 现象：文件系统、Git、认证、日志、应用控制通道均集中注册。
- 风险：虽然有 preload 白名单，但主进程仍是高价值攻击面。
- 验证：检查白名单与实际处理函数是否一一对应。

#### 3.2 `app:openExternal` 仅做协议白名单，未做更细粒度域名控制
- 位置：`electron/main.js`
- 现象：只允许 `http/https`。
- 风险：能防止明显危险协议，但仍允许任意外链打开。
- 验证：测试非 http(s) 链接必须被拒绝。

#### 3.3 `app:getE2EState` 暴露运行状态用于测试
- 位置：`electron/main.js`
- 现象：用于 E2E 的状态查询接口暴露在 IPC 层。
- 风险：设计上是可接受的，但需要保证仅返回非敏感信息。
- 验证：返回内容是否只包含 rootDir / bool 状态等低敏信息。

### 4. 内存泄漏

#### 4.1 监控窗口、主窗口都依赖 closed 回调释放引用
- 位置：`electron/main.js`
- 现象：通过 `closed` 回调将 `win` / `monitorWin` 置空。
- 风险：如果未来窗口创建流程再扩展，需继续保持一致释放。
- 验证：反复打开关闭窗口，观察引用是否及时释放。

#### 4.2 前端与主进程均存在多处事件监听
- 位置：`src/App.tsx`、`electron/main.js`
- 现象：hashchange、IPC on/off、Electron window events 较多。
- 风险：当前均有释放路径，但属于高关注区。
- 验证：反复切换页面和窗口后，监听器数量不应增加。

### 5. 体验问题

#### 5.1 监控页与主窗口模式切换依赖 hash，用户可见性弱
- 位置：`src/App.tsx`、`electron/main.js`
- 现象：监控窗口通过 hash 打开，主窗口隐藏于根组件逻辑内。
- 风险：用户可感知到，但开发者在排查时不够直观。
- 验证：从主窗口打开监控窗口，检查切换体验。

#### 5.2 退出与切换工作目录前的保存提示是阻断式交互
- 位置：`electron/main.js`
- 现象：有 dirty 状态时会阻止并弹框确认。
- 风险：体验是合理的，但需要确保消息文案清晰。
- 验证：在 dirty 状态下尝试退出/切换工作目录。

### 6. 工程化问题

#### 6.1 Electron 主进程仍然是 JS，实现与前端 TS 分离
- 位置：`electron/main.js`
- 现象：主进程 JS 文件较长。
- 风险：类型约束不足，维护难度偏高。
- 验证：检查构建是否稳定。

#### 6.2 `before-quit`、窗口创建、菜单构建等职责集中于单文件
- 位置：`electron/main.js`
- 现象：功能分区明确，但文件本身很长。
- 风险：后续再加功能容易继续膨胀。
- 验证：阅读文件结构即可感知。

---

### 7. TS 类型问题

#### 7.1 主进程 JS 缺少类型保护
- 位置：`electron/main.js`
- 现象：所有逻辑是 JS，IPC 回调、返回值、窗口状态都无 TS 保护。
- 风险：改动时容易引入低级错误。
- 验证：新增 IPC / 修改返回值时是否容易遗漏。

#### 7.2 前端与主进程通信的返回结构没有统一类型定义约束到主进程
- 位置：`electron/main.js`、`src/types/electron-api.ts`
- 现象：前端有 API 类型，但主进程处理仍松散。
- 风险：前后端接口易发生漂移。
- 验证：对照前端声明与主进程实现。

## 二、补充的架构优化目标

- 保持主窗口、监控窗口与退出保存流程稳定
- 保持 Electron 通信白名单安全
- 减少主进程单文件继续膨胀
- 持续收口 hash 路由式的隐式窗口识别
