## 一、发现的问题清单

### 1. 功能问题

#### 1.1 `GraphPage` 仍然存在页面级状态编排与业务事件耦合
- 位置：`src/components/GraphPage.tsx`
- 现象：页面仍直接负责右侧面板 tab 切换、边右键后进入样式页、节点删除入口、快捷键入口等多个交互协调。
- 风险：页面后续再加功能时容易再次膨胀，复杂交互难以追踪。
- 复现/验证：打开图谱页，触发节点删除、边右键、面板切换、快捷键操作，检查多个交互是否在页面层集中处理。

#### 1.2 右键边时样式页与选中边状态依赖页面层同步
- 位置：`src/components/GraphPage.tsx`、`src/components/GraphCanvas.tsx`
- 现象：`GraphCanvas` 只抛事件，`GraphPage` 决定是否切换到样式页并设置 `selectedEdgeId`。
- 风险：若未来其它入口也能修改边选中态，容易出现状态源分叉。
- 复现/验证：右键边，检查右侧是否切到样式页，边 ID 是否同步选中。

#### 1.3 `GraphCanvas` 仍保留 UI 之外的页面行为语义
- 位置：`src/components/GraphCanvas.tsx`
- 现象：画布层仍处理边/空白点击后的状态清理入口，并直接调用全局 store 的清选操作。
- 风险：画布层并非纯渲染组件，职责边界仍有残留。
- 复现/验证：点击画布空白区、双击空白区、右键边，检查清理与切换行为是否由页面层统一决定。

#### 1.4 `useGraphPageController` 使用快照式导航状态
- 位置：`src/hooks/useGraphPageController.ts`
- 现象：`nav = getNavState()` 依赖 render 快照，页面副作用根据快照触发。
- 风险：高频状态变化时，容易出现“下一次 render 才更新”的认知偏差。
- 复现/验证：快速切换 Tab、搜索词、选中态，观察 controller effect 是否始终与最新状态一致。

### 2. 逻辑 Bug

#### 2.1 `useKeyboard` 的 Escape 语义仍以全局清选为主
- 位置：`src/hooks/useKeyboard.ts`
- 现象：Escape 仅调用 `clearSelection()` 和关闭菜单，不再显式调用导航上下文清理接口。
- 风险：当前功能可用，但若未来 `useNavContext` 的选中态语义增强，快捷键层可能无法覆盖全部业务状态。
- 复现/验证：图谱页按 Escape，检查节点、边、右键菜单、右侧面板是否都被正确清理。

#### 2.2 `useGraph` 的选中态写入仍是双源兼容实现
- 位置：`src/hooks/useGraph.ts`、`src/hooks/useNavContext.ts`
- 现象：Tab 模式写 `tabStore`，非 Tab 模式写 `appStore`。
- 风险：不是立刻出错，但如果后续新增入口，没有严格遵守写入规范，就可能出现状态不一致。
- 复现/验证：分别在 Tab 模式和非 Tab 模式选择节点、删除节点、切换页面，观察状态是否一致。

#### 2.3 `useGraph/navigation.ts` 仍依赖底层 store 的历史栈语义
- 位置：`src/hooks/useGraph/navigation.ts`
- 现象：导航模块内直接调用 `tabStore` / `roomStore`，并根据 `tabId` 分叉。
- 风险：语义是正确的，但重复模式较多，未来更易引入分支误差。
- 复现/验证：执行返回上一层、历史跳转、回根级，检查是否与 Tab/全局模式一致。

### 3. 安全风险

#### 3.1 `electron/preload.js` 仍为传统 JS 写法，安全策略依赖白名单正确性
- 位置：`electron/preload.js`
- 现象：通过白名单限制 IPC 通道，整体思路正确，但为纯 JS 文件，缺少类型约束。
- 风险：新增通道时可能忘记同步白名单或误放行，造成 IPC 暴露风险。
- 复现/验证：尝试在渲染进程调用未授权通道，确认是否被拦截。

#### 3.2 预加载层对 `send`/`sendSync` 的通道策略偏保守但不对称
- 位置：`electron/preload.js`
- 现象：`send` 和 `sendSync` 共用同一白名单集合。
- 风险：将来如果某些通道只允许异步/同步其中一种，容易误配置。
- 复现/验证：检查所有 IPC 调用是否只通过白名单访问。

#### 3.3 `App.tsx` 直接依赖窗口 hash 控制监控页
- 位置：`src/App.tsx`
- 现象：通过 `window.location.hash === '#/monitor'` 切换监控页。
- 风险：目前能工作，但路由逻辑是隐式的，后续若加入更多 hash 场景，可能出现冲突。
- 复现/验证：在不同 hash 下切换页面，确认不会误进入监控页。

### 4. 内存泄漏

#### 4.1 需要持续关注全局事件监听清理
- 位置：`src/App.tsx`、`src/hooks/useKeyboard.ts`、`src/hooks/useGraphPageController.ts`
- 现象：当前代码中有多个全局监听/订阅入口，整体上都有清理，但属于高风险区域。
- 风险：若后续继续增加监听器，容易遗漏 `off/removeEventListener`。
- 复现/验证：反复挂载/卸载页面，确认监听器没有叠加。

#### 4.2 Blob URL 统一回收依赖应用卸载
- 位置：`src/App.tsx`、`src/core/storage.ts`
- 现象：应用卸载时回收 Blob URL，设计是正确的。
- 风险：若中途异常退出或新路径未纳入回收，仍可能造成资源泄漏。
- 复现/验证：切换/关闭多个带图片的卡片后，检查 Blob URL 是否持续增长。

### 5. 体验问题

#### 5.1 `GraphPage` 仍偏厚，阅读成本高

#### 5.3 边右键行为需要依赖页面层联动
- 位置：`src/components/GraphPage.tsx`、`src/components/GraphCanvas.tsx`
- 现象：右键边后要页面层决定 tab 切换和边选中。
- 体验影响：当前能用，但若将来重构回调链，容易断联。

## 四、功能优化详情

### 问题 1：GraphPage 仍承担过多页面编排职责
- 问题描述：页面层仍直接协调多种交互。
- 原因分析：拆分后的边界已变清晰，但页面仍是最终聚合点。
- 解决方案：后续只保留必要的布局与事件编排，避免继续回灌业务逻辑。
- 验证方式：检查页面是否只做组合和事件转发。

### 问题 2：右键边依赖页面层联动样式页与选中态
- 问题描述：画布不再直接控制页面状态，但页面层仍要同步边选中信息。
- 原因分析：受控状态已经存在，但状态流还需要继续保持一致。
- 解决方案：后续统一通过页面层单点处理。
- 验证方式：右键边后检查样式页和选中态是否同步。

### 问题 3：IPC 白名单配置依赖人工维护
- 问题描述：preload 白名单是核心安全边界。
- 原因分析：纯 JS 文件缺少类型约束。
- 解决方案：后续对新增 IPC 做白名单审查并考虑类型化。
- 验证方式：尝试未授权通道调用必须被拒绝。
