## 一、发现的问题清单

### 1. 功能问题
1. `App.tsx` 通过 hash 控制监控页，路由语义隐式，后续扩展容易冲突。
2. `GraphPage` 仍有页面编排职责残留，虽然已拆出 `GraphCanvas` / `GraphRightPanel`，但页面仍负责多个业务协调入口。
3. `GraphCanvas` 的空白区点击、边右键、节点右键等交互依赖页面层联动，功能完整但职责仍偏重。
4. `tabStore` / `roomStore` 的双源导航结构可用，但 Tab 与全局模式的语义桥接仍需持续关注。
5. `storage.ts` 中布局、图片、配置、卡片读写聚合在单文件，功能完整但边界较重。

### 2. 逻辑 Bug
1. `before-quit` 中异步 flush 保存与 `app.quit()` 的状态切换依赖 `_isQuittingAfterFlush` 标记，流程可用但需要持续验证重入边界。
2. `confirmAndFlushBeforeExit()` 通过 `executeJavaScript` 调用渲染层关闭守卫，若渲染层未注入 guard 会退化成无脏状态，这属于可接受但需要确认的边界行为。
3. `tabStore.removeTab()` 关闭当前 Tab 的 fallback 逻辑依赖数组位置，行为正确但在边界场景需持续验证。
4. `roomStore.goBack()` 与 `tabStore.goBackInTab()` 在“返回全局视图”时语义接近但实现不同，未来改动容易引入分支不一致。

### 3. 安全风险
1. `electron/preload.js` 仍以手工白名单暴露 IPC，安全边界集中，新增通道时需要严格审查。
2. `send` / `sendSync` 目前共用不同层级的白名单命名和策略，后续维护时容易误读或误改。
3. `electron/main.js` 直接使用 `executeJavaScript` 访问渲染层关闭守卫，属于强耦合通道，需确保仅访问固定、已知对象。
4. `storage.ts` 的文件系统委托依赖底层 FSB 安全性，路径校验必须继续保证。

### 4. 内存泄漏
1. `App.tsx` 中存在多个全局事件监听入口（hashchange、before-quit、electronAPI 订阅），虽均有清理，但这是高风险区。
2. `storage.ts` 的防抖保存与 Blob URL 管理依赖回收链路完整，若新增调用点忘记清理可能造成泄漏。
3. `preload.js` 的监听回调包装使用 WeakMap 去重，整体方向正确，但渲染层若忘记 `off` 仍可能堆积监听器。
4. `tabStore` / `roomStore` 为长期单例状态，Tab 多且会话长时内存会持续增长，应持续监控。

### 5. 体验问题
1. `GraphCanvas` 的空白点击清理、右键边切换样式页等行为目前能用，但交互语义分散在页面和画布之间。
2. `App.tsx` 的监控窗口 hash 切换为隐式入口，用户/维护者都不易一眼理解。
3. `storage.createCard()` 遇到同名冲突时直接失败，需要上层确保友好提示。
4. `storage.ts` 中配置与布局的归一化逻辑较隐式，体验层问题往往会在数据刷新后才暴露。

### 6. 工程化问题
1. `storage.ts` 文件职责过于集中，后续扩展时维护成本会持续升高。
2. `electron/main.js`、`electron/preload.js` 仍是纯 JS，缺少编译期类型保护。
3. `package.json` 有基础 dev/build/test 脚本，但缺少更明确的统一检查入口。
4. `docs/opt_task.md` 中目录拼写为 `opt_task_repot`，与实际目录不一致，容易误导流程执行。

### 7. TS 类型问题
1. `GraphCanvas` 仍存在局部 `as any` 强转，类型保护尚未完全闭合。
2. `storage.ts` 中对配置/元数据/JSON 读写仍有较多 `unknown` 与强制断言，边界类型不够强。
3. `electron/*` 主/预加载脚本无 TS 类型检查，后续 IPC 接口变更时更易漏改。
4. `useGraph` / `useNavContext` 的双源状态接口是可用的，但类型上仍较复杂，调用方容易误解写入语义。

## 二、架构优化目标

1. 保持现有功能不回退，继续稳定 Tab / 图谱 / 保存 / Electron 通信链路。

5. 持续减少 `any` 和强制断言，提升 TS 可维护性。

## 四、功能优化详情

### 问题 1：监控页通过 hash 隐式切换
- 问题描述：`App.tsx` 使用 `#/monitor` 决定渲染 `MonitorPage`。
- 原因分析：这是一个简单有效的实现，但路由语义隐式。
- 解决方案：后续若增加更多页面，建议统一路由策略或至少集中封装 hash 解析。
- 验证方式：切换 hash，确认主窗口/监控窗口渲染符合预期。

### 问题 2：Tab / Room 双源状态仍需严格同步
- 问题描述：Tab 内导航状态与全局 room 状态仍由不同 store 维护。
- 原因分析：为了支持多 KB Tab 隔离状态，这种拆分是有意设计。
- 解决方案：继续保持统一入口调用，不要再扩散新的直连写入点。
- 验证方式：切换 Tab、钻入/返回、关闭 Tab 后检查状态一致性。

### 问题 3：storage.ts 文件职责过重
- 问题描述：配置、布局、图片、文件、卡片操作都聚合在单文件。
- 原因分析：历史迭代导致职责扩张。
- 解决方案：后续可以按“配置/布局/图片/卡片”做内部拆分，但不要破坏现有 API。
- 验证方式：确保 `readLayout/saveLayout/readMarkdown/writeMarkdown` 行为不变。
