## 一、补充发现的问题清单

### 1. 功能问题

#### 1.1 `App.tsx` 里的窗口模式切换仍偏隐式
- 位置：`src/App.tsx`
- 现象：通过 `window.location.hash === '#/monitor'` 识别监控窗口，主窗口与监控窗口共享同一渲染入口。
- 风险：当前可用，但监控模式依赖 hash，后续若扩展更多 hash 路由，可能出现冲突或误判。
- 验证：分别打开主窗口与 `#/monitor`，确认页面渲染分支正确。

#### 1.2 `App.tsx` 的 Tab 初始化与工作目录自动导航存在耦合
- 位置：`src/App.tsx`
- 现象：初始化 home tab、监听主进程导航事件、自动切 home 的流程集中在根组件中。
- 风险：根组件承担过多生命周期逻辑，后续新增启动行为时容易再次膨胀。
- 验证：启动应用、切换工作目录、重启，检查是否仍能正确落到 setup/home/graph。

#### 1.3 关闭 Tab 的保存流程依赖动态导入与全局 flush
- 位置：`src/App.tsx`、`src/core/close-guard.ts`
- 现象：关闭有脏改的知识库 Tab 时，会先确认，再动态导入 flush 方法进行保存。
- 风险：逻辑可用，但路径较绕，若后续保存链路变化，容易出现关闭阻塞或误判。
- 验证：修改知识库后关闭 Tab，确认能先保存再关闭；模拟保存失败，确认不会误关闭。

### 2. 逻辑 Bug

#### 2.1 `App.tsx` 中的 `useEffect` 依赖存在轻微不一致
- 位置：`src/App.tsx`
- 现象：有些 effect 使用了函数引用，但依赖列表不完全对应实际语义（例如 `initHomeTab` 被引入但部分 effect 实际是基于 `showHome` / `storage` 行为）。
- 风险：目前通常能正常工作，但后续重构时可能引入误触发或遗漏触发。
- 验证：启动后切换工作目录、刷新页面、检查自动导航是否重复或丢失。

#### 2.2 监控窗口打开逻辑与主窗口状态切换耦合
- 位置：`electron/main.js`
- 现象：监控窗口通过主进程菜单项打开，且使用 `#/monitor` hash 作为渲染标识。
- 风险：如果未来主窗口也引入 hash 路由，监控窗口判定可能不够稳健。
- 验证：打开/关闭监控窗口，反复切换主窗口 hash，观察是否有误入监控页。

#### 2.3 `useGraph` 仍保留双源选中态逻辑
- 位置：`src/hooks/useGraph.ts`、`src/hooks/useNavContext.ts`
- 现象：Tab 模式与非 Tab 模式写入不同 store，虽然功能可用，但逻辑上仍是双源。
- 风险：新增写入入口时，如果不严格遵守约定，会出现选中态不一致。
- 验证：在 Tab 和非 Tab 模式中分别选中、删除、切换节点，检查状态是否一致。

### 3. 安全风险

#### 3.1 `electron/preload.js` 中 `send` 与 `sendSync` 白名单合并
- 位置：`electron/preload.js`
- 现象：当前 `send` 和 `sendSync` 共用同一白名单集合，而不是分别约束。
- 风险：后续若某些通道只允许同步或异步中的一种，容易误配。
- 验证：尝试调用未在白名单的通道，应被拒绝。

#### 3.2 `electron/main.js` 中 `openExternal` 仅允许 http/https，但仍需关注 URL 注入
- 位置：`electron/main.js`
- 现象：已限制协议，但调用者仍需确保传入内容来自可信来源。
- 风险：若未来接入更多外部输入，仍需保留协议校验。
- 验证：传入非 http/https 链接应返回 false。

#### 3.3 IPC 白名单维护仍然高度依赖人工同步
- 位置：`electron/preload.js`、`electron/main.js`
- 现象：新增 IPC 需要同时改 preload 白名单和主进程 handler。
- 风险：若遗漏其中一侧，可能出现“能调用但没处理”或“有处理但不可调用”。
- 验证：新增 IPC 前后确认主进程与 preload 均更新。

### 4. 内存泄漏

#### 4.1 主窗口和监控窗口都存在多处长期监听
- 位置：`src/App.tsx`、`electron/main.js`
- 现象：窗口、hashchange、before-quit、Electron 事件都在多个地方注册。
- 风险：当前大体有清理，但后续新增监听时很容易遗漏销毁。
- 验证：多次打开关闭窗口，确认监听不会叠加。

#### 4.2 Blob URL 回收仍依赖应用卸载
- 位置：`src/App.tsx`、`src/core/storage.ts`
- 现象：Blob URL 统一在卸载回收，属于正确做法。
- 风险：异常退出或未来新增来源未纳入回收时，仍可能泄漏。
- 验证：反复切换带图片卡片，观察资源是否增长异常。

### 5. 体验问题

#### 5.1 `App.tsx` 启动态逻辑集中，阅读成本高
- 位置：`src/App.tsx`
- 现象：启动、自动导航、监控窗口判定、关闭前保存等流程都在根组件中。
- 体验影响：代码可读性偏低，维护者理解启动流程需要较长上下文。
- 验证：阅读 App 根组件，是否能迅速看出启动路径。

#### 5.2 工作目录切换后的用户反馈链路较长
- 位置：`electron/main.js`、`src/App.tsx`
- 现象：切换工作目录会先 flush、再重置主窗口、再发送 session reset。
- 体验影响：当前正确，但失败/取消路径需要足够明确，否则容易让用户误以为操作卡住。
- 验证：切换工作目录前制造未保存更改，检查提示与结果。

#### 5.3 监控页通过 hash 隐式进入
- 位置：`src/App.tsx`、`electron/main.js`
- 现象：监控页进入方式不够显式。
- 体验影响：未来用户如果手动修改 hash，可能看到意料之外的页面。
- 验证：手工修改 hash，确认渲染分支正确且不会误伤主流程。

### 6. 工程化问题

#### 6.1 `docs/opt_task.md` 存在路径拼写不一致
- 位置：`docs/opt_task.md`
- 现象：文档写的是 `docs/opt_task_repot`，实际目录是 `docs/opt_task_report`。
- 风险：后续执行报告输出时容易写错目录。
- 验证：人工核对输出目录即可。

#### 6.2 `package.json` 缺少统一 lint/check 聚合脚本
- 位置：`package.json`
- 现象：已有 dev/build/test，但缺少统一检查入口。
- 风险：后续优化和回归检查容易遗漏某一类验证。
- 验证：执行现有脚本看是否能覆盖全部检查项。

#### 6.3 Electron 代码仍以 JS 为主，缺少类型化约束
- 位置：`electron/*.js`
- 现象：主进程/预加载/服务脚本均为 JS。
- 风险：维护时缺少 TS 提示，尤其是 IPC 和路径逻辑。
- 验证：检查新增 IPC 与参数约束是否完整。

### 7. TS 类型问题

#### 7.1 前端侧仍存在少量 `any` / 强制断言痕迹
- 位置：`src/components/GraphCanvas.tsx`、`src/hooks/useGraph.ts`
- 现象：仍有部分类型断言与绕过。
- 风险：类型防护不够完整。
- 验证：查看相关文件，确认未引入新的 `any` 扩散。

#### 7.2 Electron JS 文件无法享受 TypeScript 编译期保护
- 位置：`electron/*.js`
- 现象：预加载和主进程脚本不受 TS 检查。
- 风险：路径、通道、参数签名更容易出错。
- 验证：通过运行时检查和手工测试确认无误。

## 四、功能优化详情

### 问题 1：App 根组件承载过多启动逻辑
- 问题描述：启动、自动导航、hash 判定、Tab 初始化、关闭前保存都在根组件内。
- 原因分析：全局流程都集中在入口层，导致根组件职责较重。
- 解决方案：后续可考虑把启动流程拆成更明确的初始化钩子或服务层。
- 验证方式：启动应用，确认 setup/home/graph/monitor 路径都正确。

### 问题 2：监控窗口依赖 hash 判定
- 问题描述：监控页是否渲染通过 `#/monitor` 判断。
- 原因分析：当前为了复用入口而采用隐式分流。
- 解决方案：后续若扩展路由，建议将其独立成更明确的窗口模式判定。
- 验证方式：修改 hash 后确认页面切换符合预期。

### 问题 3：IPC 白名单维护成本高
- 问题描述：preload 白名单与主进程 handler 必须同步维护。
- 原因分析：Electron 架构要求前后端同时约束。
- 解决方案：新增 IPC 时采用审查清单，并考虑未来类型化方案。
- 验证方式：调用非白名单通道必须被拒绝。
