## 一、发现的问题清单

### 1. 功能问题
1. `useGit` 功能完整，覆盖可用性检查、仓库初始化、状态、历史、远端、冲突、认证、缓存辅助等接口。
2. `GitBackend` 前端 IPC 封装完整，所有 Git 功能都能从同一入口触发。
3. `GitCache` 具备状态缓存、脏状态标记、监听器通知与过期清理功能，整体逻辑完整。

### 2. 逻辑 Bug
1. `useGit` 返回的 `onStatusChange` 依赖 `GitCache.onStatusChange`，逻辑正确，但如果调用方忘记取消订阅，仍可能残留监听。
2. `GitCache.setStatus()` 在缓存超限时会按时间清理旧项，当前策略可用，但属于近似 LRU，不是严格访问序列 LRU。
3. `GitBackend._call()` 在 IPC API 未就绪时会直接 reject，逻辑合理，但调用方必须正确处理失败。
4. `GitBackend` 所有返回值使用 `Promise<unknown>`，若上游结构变化，前端解析失败会延迟暴露。

### 3. 安全风险
1. `GitBackend` 通过 `window.electronAPI.invoke` 直接桥接主进程，安全边界依赖 preload 白名单和主进程校验。
2. Git 远端 URL、认证信息等属于敏感配置，调用链中需要持续避免日志泄露。
3. `GitCache` 通知监听器时捕获异常并记录日志，整体合理，但日志内容要避免输出敏感字段。
4. `useGit` 暴露的认证相关接口应持续保持最小暴露原则。

### 4. 内存泄漏
1. `useGit` 通过 `cleanupRefCount` 控制缓存清理定时器启停，设计正确，但需要确保组件卸载计数严格对称。
2. `GitCache.onStatusChange()` 维护全局监听器数组，若调用方不清理会持续增长。
3. `_cleanupTimer` 在最后一个使用者卸载后才停止，当前机制合理，但多入口场景仍需保持谨慎。

### 5. 体验问题
1. `useGit` 功能多，接口虽统一，但调用方需要理解多个返回方法，学习成本较高。
2. `GitCache` 的状态显示与缓存清理策略对用户是透明的，体验上可用，但若缓存过期时 UI 变化突然，需持续验证。
3. Git 认证与冲突处理功能完整，但上层提示文案要保持友好，否则会影响整体体验。

### 6. 工程化问题
1. `useGit` 是较厚的 Git 统一 hook，适合继续作为稳定入口，但后续若功能更多，接口会继续膨胀。
2. `GitBackend` 返回值目前较多 `unknown`，后续若要提升工程化质量，建议补强具体类型。
3. `GitCache` 状态通知、清理、缓存阈值等逻辑都集中在一个文件，后续维护时要重点关注。

### 7. TS 类型问题
1. `GitBackend` 所有方法返回 `Promise<unknown>`，类型保护弱。
2. `GitCacheStatus` 虽有基础字段，但状态对象整体仍较宽松。
3. `useGit` 中 `authSetType`、`diff`、`log` 等参数使用 `object` 或 `string` 宽泛类型，后续可逐步收窄。

## 四、功能优化详情

### 问题 1：GitCache 采用近似 LRU 的过期清理策略
- 问题描述：缓存超限时按 timestamp 清理旧项。
- 原因分析：实现简单，能满足当前需求。
- 解决方案：当前可接受；若后续要求更精确，再考虑严格 LRU。
- 验证方式：连续查询多个仓库后检查缓存是否正常命中/过期。

### 问题 2：useGit 返回接口较厚
- 问题描述：统一 hook 暴露了很多方法。
- 原因分析：Git 功能本身就较多。
- 解决方案：保持稳定入口，后续以类型精炼为主，不轻易拆分行为。
- 验证方式：仓库初始化、状态查询、冲突处理、认证流程仍正常。

### 问题 3：GitBackend 返回值宽泛
- 问题描述：大量 Promise<unknown>。
- 原因分析：前端与主进程返回结构尚未完全收口。
- 解决方案：后续逐步补充具体类型定义。
- 验证方式：功能调用结果在 UI 中正确解析。
