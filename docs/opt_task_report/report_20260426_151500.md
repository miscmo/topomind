## 一、发现的问题清单

### 1. 功能问题
1. `PromptModal` 能替代 `window.prompt()`，功能完整，但默认值同步逻辑依赖 `defaultValue` 与 `visible` 变化，若后续增加多次打开同一个弹窗的复杂流程，需确认输入值不会残留。
2. `ConfirmModal` 功能完整，支持确认/取消，但长文本场景下尚未看到特别的滚动/截断处理。
3. `MonitorPage` 功能完整，日志查询/筛选/展示链路基本闭合，但作为监控页，后续若要更强的过滤和采样展示，可能需要拆分更多子模块。
4. `HomePage` / `SetupPage` 功能完整，分别负责首页与初始化流程，但与 `App.tsx` 的切换逻辑仍然耦合在根组件。

### 2. 逻辑 Bug
1. `PromptModal` 点击遮罩会关闭，输入框 `Escape` 也会关闭，逻辑正确；但 `Enter` 直接确认，若输入多行或 IME 场景较复杂，需持续验证输入体验。
2. `ConfirmModal` / `PromptModal` 依赖 store 的 open/close 行为，若调用方忘记传正确的回调或默认值，弹窗会表现为“能开但回填不对”。
3. `MonitorPage` 查询筛选依赖日志 store / service 的返回结构，若日志字段变化，筛选结果可能偏差。
4. `useStorage` 作为渲染层统一存储入口，逻辑本身合理，但所有文件系统操作都汇聚于此，调用方出错时更难第一时间定位。

### 3. 安全风险
1. `PromptModal` 会把用户输入回传到上层，若上层将输入直接用于路径/命令拼接，存在注入风险；需要调用方严格校验。
2. `MonitorPage` 展示日志内容，若日志中出现敏感信息，需要确保日志采集层已经做过脱敏。
3. `useStorage` 统一调用 IPC，安全边界依赖 preload 白名单和主进程校验。
4. `SetupPage` 在创建/切换工作目录时涉及文件操作，路径合法性必须由底层持续校验。

### 4. 内存泄漏
1. `PromptModal` / `ConfirmModal` 本身没有明显定时器或订阅，但作为全局挂载弹窗，需要继续关注 store 订阅清理。
2. `MonitorPage` 若使用轮询或订阅刷新日志，需要确保卸载时清理。
3. `useStorage` 内部如果缓存 blob URL 或防抖定时器，需确保调用链能回收。

### 5. 体验问题
1. `PromptModal` 的默认值回填可用，但用户多次打开时若不清空状态，可能有残留感。
2. `ConfirmModal` 若文案较长，当前布局需要确认是否足够适配。
3. `MonitorPage` 作为性能/日志页，当前主要功能可用，但筛选与时间轴体验可继续优化。
4. `SetupPage` 初始化/切换工作目录流程对新用户较关键，当前功能完整但可继续加强错误提示。

### 6. 工程化问题
1. `PromptModal` 和 `ConfirmModal` 都是全局挂载型模态组件，随着功能增多需要保持 store 协议稳定。
2. `MonitorPage` 属于独立子系统，后续如果增加更多监控维度，建议拆更多子组件而不是继续堆逻辑。
3. `useStorage` 文件偏大，承担过多存储入口职责，后续需要继续关注模块边界。
4. `App.tsx`、`HomePage`、`SetupPage` 的启动链路是核心工程入口，变更需格外谨慎。

### 7. TS 类型问题
1. `PromptModal` 的事件处理使用的是 React.KeyboardEvent，类型可用但在复杂输入法场景中仍需谨慎。
2. `MonitorPage` 与日志服务之间的数据结构若变化，类型联动要同步检查。
3. `useStorage` 涉及多种返回结构，类型声明较多，后续若继续扩展建议适当提纯接口。

## 四、功能优化详情

### 问题 1：PromptModal 默认值同步依赖 visible/defaultValue
- 问题描述：弹窗打开/默认值变化时回填输入值。
- 原因分析：当前做法可用，但在复杂重复打开场景下需要确认不会残留旧值。
- 解决方案：后续若发现残留，可在 open/close 协议里显式重置。
- 验证方式：连续多次打开同一 prompt，检查输入值是否正确回填。

### 问题 2：MonitorPage 作为独立子系统后续可继续拆分
- 问题描述：监控页功能完整，但模块职责可能继续增长。
- 原因分析：监控页天然是多功能展示页。
- 解决方案：未来如果增加过滤、图表、时间轴，建议拆子组件。
- 验证方式：打开监控页后检查查询与展示是否正常。

### 问题 3：useStorage 文件偏大
- 问题描述：统一存储入口承担了太多职责。
- 原因分析：长期迭代导致的聚合。
- 解决方案：后续可按“布局/图片/配置/卡片/KB”拆内部 helper。
- 验证方式：所有文件读写与 IPC 调用仍需保持原功能。
