## 一、发现的问题清单

### 1. 功能问题
1. `NavTree` 负责知识库列表加载与“新建知识库”入口，功能完整，但仍直接操作 `roomStore` 与 `appStore`，页面语义耦合较重。
2. `Breadcrumb` 通过 model/actions 分层后结构清晰，功能完整；但历史项索引换算依赖 `items[0]` 为 root 的约定，未来变更时容易出错。
3. `DetailPanel` 同时承担 Markdown 读取/保存、重命名、删除、子概念跳转、保存守卫注册等功能，功能完整但职责偏重。
4. `useGraphPageController` 现有逻辑完整，页面日志、房间加载、搜索高亮、dirty 同步、tab 保存注册都能覆盖，但依赖 render 快照，后续高频状态变更时需持续验证。

### 2. 逻辑 Bug
1. `DetailPanel` 中“子概念标签点击进入房间”直接调用 `roomStore.enterRoom()`，未经过页面/导航统一入口，Tab 模式下可能导致状态桥接语义不够一致。
2. `DetailPanel` 删除节点后直接 `collapseRightPanel()`，这是当前预期行为，但如果未来要支持“删除后保留详情页”的交互，需修改此逻辑。
3. `NavTree` 新建知识库后会重新拉取列表并触发 KB 刷新，流程可用，但如果 `createKB` 失败后的提示不够明确，用户会误以为创建成功。
4. `Breadcrumb` 的根节点与历史项点击逻辑是正确的，但索引计算强依赖渲染顺序，属于可维护性风险点。

### 3. 安全风险
1. `DetailPanel` 使用 `dangerouslySetInnerHTML` 渲染 Markdown，虽然已通过 DOMPurify 清理，但仍是 HTML 注入高风险点，需要确保净化逻辑始终生效。
2. `NavTree` 通过 `prompt` 接收知识库名称，若上层校验不足，可能引入非法名称或路径问题。
3. `DetailPanel` 调用 `registerTabSaver` 保存 Markdown，若保存守卫逻辑被误绕过，可能造成未保存内容丢失。
4. `Breadcrumb`、`NavTree`、`DetailPanel` 均通过日志上报关键路径，日志内容需保持不包含敏感信息。

### 4. 内存泄漏
1. `DetailPanel` 使用两个 requestSeq ref 防止异步竞态，整体方向正确；但如果后续引入更多异步读取，仍需保持“只响应最新请求”的模式。
2. `NavTree` 在刷新知识库列表时会重新 setState，当前没有明显泄漏，但如果未来增加轮询或订阅，要注意清理。
3. `Breadcrumb` 基于 model/actions 的拆分方式较清晰，但如果后续增加监听，需要注意不要在组件内部累积订阅。

### 5. 体验问题
1. `DetailPanel` 内容较多，虽然功能完整，但用户在编辑、重命名、删除、查看子概念之间切换时需要较多步骤。
2. `NavTree` 的新建知识库按钮文案为“新建卡片”，与实际功能“新建知识库”不完全一致，容易造成理解偏差。
3. `Breadcrumb` 需要依赖历史链与 root 约定，当前使用体验正常，但对维护者来说理解成本较高。
4. `DetailPanel` 的空状态和加载状态目前可用，但若后续引入更复杂的异步读写，建议补更明确的 loading/error 反馈。

### 6. 工程化问题
1. `DetailPanel` 单文件职责偏重，后续继续增长会增加维护成本。
2. `NavTree` 直接依赖 `useStorage` 拉取 KB 列表与创建 KB，功能可用，但如果后续加入更复杂的筛选/分页，可能需要进一步抽象。
3. `Breadcrumb` 的注释已较清晰，但历史索引规则属于隐式约定，建议持续保留文档说明。
4. `DetailPanel` 内部的异步请求控制采用 seq ref，工程上可接受，但如果未来迁移到更复杂的加载模型，建议封装为统一数据加载工具。

### 7. TS 类型问题
1. `DetailPanel` 中存在较多联合类型/断言处理 Markdown 与卡片目录结果，类型安全可用但不够强。
2. `NavTree` / `Breadcrumb` 依赖的类型来自多个 `src/types` 导出，类型层面没有明显错误，但调用链较长。
3. `useGraphPageController` 的导航快照结构较窄，如果后续扩展更多导航字段，类型需要同步扩展。

## 四、功能优化详情

### 问题 1：DetailPanel 职责偏重
- 问题描述：一个组件承担 Markdown 读写、重命名、删除、子概念跳转、保存守卫等多个功能。
- 原因分析：这是当前业务需要，但随着功能增长，单文件复杂度会持续升高。
- 解决方案：后续可考虑拆分为显示/编辑/操作三层，但需保持行为一致。
- 验证方式：编辑、保存、重命名、删除、进入子概念都应继续正常工作。

### 问题 2：NavTree 新建按钮文案与行为不完全一致
- 问题描述：按钮文案为“新建卡片”，实际行为是“新建知识库”。
- 原因分析：历史文案未同步更新。
- 解决方案：后续统一文案，避免用户理解歧义。
- 验证方式：点击按钮后的行为与文案描述一致。

### 问题 3：Breadcrumb 历史索引规则隐式
- 问题描述：历史索引计算依赖 root 位置约定。
- 原因分析：当前实现能工作，但约定隐藏在代码逻辑中。
- 解决方案：后续保留注释或将计算逻辑封装为 helper。
- 验证方式：点击历史项与 root 均能跳转正确。
