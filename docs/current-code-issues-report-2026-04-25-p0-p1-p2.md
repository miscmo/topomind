# 当前代码问题整改任务清单（2026-04-25）

> 基于《当前代码问题复盘报告（2026-04-25）》整理，目标是把现有问题拆成可以直接交给 Claude Code 实现的任务。

## 总体判断

你这版代码已经比上一版更模块化，尤其是：

- `Breadcrumb` 拆出了 model / actions
- `useGraph` 拆出了 `graphBuilder` / `graphOperations`

但核心问题仍在，说明当前最需要做的不是继续堆功能，而是**收敛单一事实来源、拆分控制器职责、清理兼容层**。

---

## P0 必须先做

### P0-1 统一导航与选中状态的单一事实来源

**目标**

让当前路径、搜索词、选中节点、当前 tab/room 状态有明确的唯一来源，避免 app store、tab store、room store 之间来回双写。

**需要处理的点**

- 明确 `selectedNodeId` 的真值来源
- 明确 `searchQuery` 的真值来源
- 明确 `roomPath` / `kbPath` / `currentRoomPath` / `currentKBPath` 的语义
- 收敛 `GraphPage` 中的兼容同步逻辑
- 收敛 `useNavContext` 中的兼容分流逻辑

**验收标准**

- 同一个状态在任意时刻只有一个写入源
- 不再需要“写完 tab 再同步 app store”的桥接式逻辑
- 切换 tab / 非 tab 时状态表现一致且可解释

---

### P0-2 继续拆分 `useGraph` 的控制器职责

**目标**

把 `useGraph` 从“全功能控制器”拆成更清晰的组合层。

**需要处理的点**

- 把导航逻辑从 `useGraph` 进一步抽出
- 把持久化逻辑从 `useGraph` 进一步抽出
- 把 React Flow 事件处理和状态同步拆开
- 让 load / save / navigate / selection / layout 各自有更清楚的边界

**验收标准**

- `useGraph` 不再同时承担状态、IO、导航、视图四类职责
- 单个函数的阅读范围显著缩小
- 新增导航策略时不需要同时改大量 unrelated 逻辑

---

### P0-3 清理明显无效、重复或误导性的分支

**目标**

去掉迁移期遗留的“看起来有保护、实际上不生效”的逻辑。

**需要处理的点**

- `navigateToRoom` 中可疑的重复条件判断
- 导航/保存中的冗余变量和无效分支
- 与当前语义不一致的历史遗留保护逻辑

**验收标准**

- 没有明显永远不成立的条件判断
- 不再保留误导维护者的“假保护”代码
- 代码逻辑更直观，维护成本更低

---

## P1 尽快推进

### P1-1 把 `GraphPage` 变成更薄的页面编排层

**目标**

让 `GraphPage` 只做页面组合和少量编排，不再承载大量副作用和状态桥接。

**需要处理的点**

- 把搜索同步、选中同步、房间恢复、日志埋点进一步外移
- 把右侧面板拖拽逻辑单独抽出去
- 把 tab / global 的条件分支减少到最少

**验收标准**

- `GraphPage` 主要负责布局和子组件组合
- 页面层副作用数量显著减少
- 后续加功能不会继续把页面撑大

---

### P1-2 统一导航领域模型命名

**目标**

减少相似字段名造成的语义混乱。

**需要处理的点**

- 统一 `roomPath` / `kbPath` / `currentRoomPath` / `currentKBPath`
- 统一 `selectedNodeId` / `selectedEdgeId` 的语义边界
- 统一 `NavState`、`BreadcrumbState`、tab state 的字段定义方式

**验收标准**

- 各模块对同一个概念的定义一致
- 新人阅读时不会被相似字段名绕晕
- 类型可以更明确地表达领域含义

---

### P1-3 让 `useNavContext` 收敛成真正的只读导航适配层或真正的导航服务

**目标**

避免这个 hook 继续变成“兼容逻辑聚合点”。

**需要处理的点**

- 明确它到底是只读数据适配器，还是带写操作的导航服务
- 如果只读，就去掉写操作
- 如果是服务，就把行为和数据模型都定义清楚

**验收标准**

- hook 的职责单一
- 不再混合“取数据”和“写状态”两类完全不同的能力
- 后续调用方更容易理解和替换

---

### P1-4 降低 app store 与 tab store 的双写

**目标**

减少跨 store 的同步成本和竞态风险。

**需要处理的点**

- 选中状态不要两边同时写
- 搜索词不要两边同时写
- 右侧面板状态不要在多个层级重复协调

**验收标准**

- 每个状态只在一个地方更新
- 其他层仅消费，不再回写
- store 之间的同步逻辑明显减少

---

## P2 中期优化

### P2-1 补齐关键路径测试

**目标**

确保后续重构不会回退。

**建议覆盖**

- tab / 非 tab 导航分支
- 历史跳转 index 映射
- root 返回行为
- 选中节点同步行为
- dirty 状态切换

**验收标准**

- 核心导航流程有单测保护
- 关键重构时可以快速判断有没有回归

---

### P2-2 把兼容层收敛成明确的迁移策略

**目标**

避免“兼容代码永久化”。

**需要处理的点**

- 明确哪些同步逻辑是临时的
- 明确哪些模块是迁移过渡层
- 明确何时可以删除旧字段或旧桥接

**验收标准**

- 兼容层有边界、有期限
- 不再无止境地叠加新桥接逻辑

---

### P2-3 继续抽象领域对象和纯计算逻辑

**目标**

让复杂流程尽量落在可测试、可复用的纯逻辑上。

**建议方向**

- 将路径推导抽成纯函数
- 将历史导航映射抽成纯函数
- 将 tab/global 的状态组合抽成统一 selector

**验收标准**

- 更多逻辑可以不依赖 React 和 store 直接测试
- 组件和 hook 更轻、更稳定

---
